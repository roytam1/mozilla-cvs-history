自分の証明書を取得する

証明書とは、クレジット カードや免許証のようなもので、インターネットや他のネットワーク上で身分を証明するために使用できる ID です。一般的な個人の身分証明書と同じように、証明書は通常、それを発行することが認められた組織によって発行されます。証明書を発行する組織は、認証局（CA）と呼ばれます。

公共の認証局、システム管理者、所属組織の特殊な認証局、または名前とパスワードよりも信頼できる証明手段を必要とする特別なサービスを提供する Web サイトから自分の身元を証明する証明書を手に入れることができます。

運転免許証の条件が運転する車種によって異なるように、証明書を手に入れる条件もその用途によって異なります。場合によっては、証明書の取得は非常に簡単で、Web サイトにアクセスし、個人情報を入力して、ブラウザに証明書をダウンロードするだけで済みます。 そうでない場合は、もう少し複雑な手順を踏まなければなりません。

現在では、認証局の URL を入力し、画面の指示に従うだけで、証明書を入手することができます。認証局の一覧については、「Client Certificates（クライアント証明書）」を参照してください。

証明書を取得したら、自動的にセキュリティ デバイスに保存されます。 お使いのブラウザには、ソフトウェア セキュリティ デバイス が組み込まれています。セキュリティ デバイスは、スマート カードのようなハードウェアの場合もあります。

運転免許証やクレジット カードのように、証明書も大切な身分証明書の 1 つであるため、他人の手に渡ると悪用されることもあります。自分の身分を証明する証明書を取得したら、バックアップをとり、マスター パスワードを設定して保護してください。

証明書を初めて取得したら、バックアップをとるよう指示されます。マスター パスワードをまだ設定していない場合は、作成するよう求められます。

証明書のバックアップおよびマスター パスワードの設定について詳しくは、「身分証明書」を参照してください。

Web ページのセキュリティを確認する

Web ページの表示中に、ウィンドウの右下に表示されるロック アイコンを見れば、コンピュータに送信されているページのコンテンツ全体が暗号化によって保護されているかどうかが分ります。

  ロック アイコンが閉じている場合、そのページは受信時に暗号化で保護されていたことを意味します。
   
  ロック アイコンが開いている場合、そのページは受信時に暗号化で保護されていなかったことを意味します。
   
  ロック アイコンが壊れている場合は、表面上 HTML ページが暗号化されていても、受信時にそのページのエレメントの一部またはすべてが暗号化されていなかったこと意味します。

ページ受信時の暗号化ステータスについて詳しくは、ロック アイコンをクリックするか、［表示］メニューから［ページ情報］を選択し、［セキュリティ］タブをクリックしてください。

［ページ情報］の［セキュリティ］タブでは、2 種類の情報を得ることができます。

  • 上部には、その Web サイトの識別情報を確認できたかどうかが示されます （証明書の確認については、「確認機能を制御する」を参照してください）。
  • 下部には、表示しているページのコンテンツがネットワークを介して送信されるときに暗号化されていたかどうかが示されます。

重要：ロック アイコンは、ページ受信時の暗号化ステータスを表すだけのものです。 暗号化されていない情報を受送信する前にその旨を知らせる通知が表示されるようにするには、適切な SSL 警告オプションを選択してください。 詳しくは、「プライバシーとセキュリティの設定 - SSL」を参照してください。

認証局を証明する証明書を管理する

一般的な身分証明書と同じように、証明書は、それを発行することが認められた組織によって発行されます。証明書を発行する組織は、認証局(CA)と呼ばれます。認証局（CA）の身元を証明する証明書は、認証局証明書と呼ばれます。

証明書マネージャは、一般に多数の認証局証明書をファイルとして持っています。証明書マネージャは、これらの認証局証明書を保持することで、対応する認証局から発行された証明書を認識および操作することができます。ただし、このリストに認証局証明書があるからといって、発行された証明書が信頼できるとは限りません。自分でまたはシステム管理者が、セキュリティの必要性に基づいて、どの証明書が信頼できるかを判断する必要があります。

ブラウザで使用できる認証局証明書をすべて表示するには、［証明書マネージャ］ ウィンドウの一番上にある［認証局］タブをクリックします。

［編集］ボタンをクリックすると、各証明書の信頼設定を表示および管理できます。認証局証明書の信頼設定を使って、認証局から発行された証明書のうち、どの種類の証明書が信頼できるかを指定できます。

スマート カードやその他のセキュリティ デバイスを管理する

スマート カードは、一般に、マイクロプロセッサを搭載したクレジット カード程度の小さなデバイスで、日本では IC カードと呼ばれています。 スマート カードには、秘密キーや証明書などの暗号情報が保存され、暗号処理を実行する機能が備わっています。

スマート カードを使用できるように、通常、コンピュータにはスマート カード リーダー（ハードウェア）が接続されており、リーダーを管理するソフトウェアがインストールされている必要があります。

スマート カードは、セキュリティ デバイスの 1 つです。セキュリティ デバイス（トークンとも呼ばれる）は、ユーザの身元に関する情報を暗号化して保存しておくハードウェアまたはソフトウェアです。デバイス マネージャを使って、スマート カードおよびその他のセキュリティ デバイスを操作します。

セキュリティ デバイスとモジュールについて

デバイス マネージャは、使用可能なセキュリティ デバイスをウィンドウに表示します。デバイス マネージャを使って、スマート カードなどの PKCS（Public Key Cryptography Standard） #11 をサポートするセキュリティ デバイスを管理できます。

PKCS #11 モジュール（セキュリティ モジュールとも呼ばれる）は、ソフトウェア ドライバがプリンタやモデムなどの外部デバイスを制御するのと同じように、いくつかのセキュリティ デバイスを制御します。スマート カードをインストールする場合は、お使いのコンピュータにスマート カード用の PKCS #11 モジュールをインストールし、それをスマート カード リーダーに接続する必要があります。

デバイス マネージャは、3 つのセキュリティ デバイスを管理する 2 つの内蔵 PKCS #11 モジュールを標準で制御します。

  • Netscape Internal PKCS #11 Module：2 つのセキュリティ デバイスを管理します。
    • 一般暗号サービス：Netscape Internal PKCS #11 Module が要求する暗号化をすべて行う特別なセキュリティ デバイスです。
    • ソフトウェア セキュリティ デバイス：外部セキュリティ デバイスに保存されていない証明書とキーを保存します。これには、ブラウザに付属の証明書のほか、自分がインストールした認証局証明書が含まれます。
  • Builtin Roots Module：Builtin Object Token と呼ばれる特殊なセキュリティ デバイスを制御します。 このセキュリティ デバイスは、ブラウザに標準の認証局証明書を保存します。

デバイス マネージャを使用すると、セキュリティ デバイスの操作を実行できます。 デバイス マネージャを開くには、以下の手順で操作します。

  3. ［証明書］パネルで、［セキュリティ デバイスの管理］をクリックします。

デバイス マネージャでは、使用可能な PKCS #11 が太字で示され、モジュール名の下に各モジュールが管理するセキュリティ デバイスが表示されます。

これらの操作は、ほとんどのセキュリティ デバイスに対して行うことができます。 ただし、Builtin Object Token や Generic Crypto Services については行えません。これらのデバイスは、常に使用可能にしておく必要がある特別なデバイスです。

詳しくは、「デバイス マネージャ」を参照してください。

SSL 警告と設定を管理する

SSL（Security Sockets Layer）プロトコルを使用すると、暗号化された情報をインターネット上の別のコンピュータと交換することができます。転送中の情報はスクランブルされているため、誰も解読することができません。SSL は、証明書によって、インターネット上でコンピュータの身元を証明するためにも使われます。

TLS（Transport Layer Security）プロトコルは、SSL に基づく新しい規格です。ブラウザは標準で SSL と TLS の両方をサポートしています。このため、ブラウザは、どのバージョンの SSL または TLS でもサポートするインターネット上のおおよそのソフトウェアと動作するため、ほとんどの人には問題がありません。

ただし、場合によっては、システム管理者または知識が豊富なユーザが、特別なセキュリティの必要性や古いソフトウェアのバグ修正のために SSL 設定を調整しなければならないこともあります。

自分でその方法をよく知っているか、または誰か設定をよく知っている人が援助してくれるのでない限り、ブラウザの SSL 設定は変更しないようにしてください。どうしても変更する必要がある場合は、以下の手順で操作してください。

  1. ［編集］メニューから［設定］を選択します。
  2. ［プライバシーとセキュリティ］カテゴリの［SSL］を選択します（サブカテゴリが表示されていない場合は、［プライバシーとセキュリティ］カテゴリをダブルクリックしてリストを展開表示します）。

詳しくは、「SSL 設定」を参照してください。

確認機能を制御する

「自分の証明書を取得する」セクションで説明したように、証明書は、運転免許証のような身分証明書で、インターネットやその他のネットワーク上で自分の身元を証明するために使用します。 ただし、証明書も運転免許証と同様に、期限が切れたり無効になることがあります。したがって、ブラウザ ソフトウェアでは、与えられた証明書を信頼する前に、何らかの方法で有効性を確認する必要があります。

このセクションでは、証明書マネージャを使って証明書を確認し、そのプロセスを管理する方法について説明します。 このプロセスを理解するには、公開キー暗号化について多少の知識が必要です。証明書の使い方をよく知らない場合は、ブラウザの証明書確認の設定を変更する前に、システム管理者に相談してください。

このセクションの内容：

確認機能のしくみ

証明書マネージャが保存している証明書を使用または表示するときは、いくつかの手順によって証明書を確認します。最低でも、証明書に記された認証局のデジタル署名は、次の条件を備えた認証局によって作成されていなければなりません。1. 認証局証明書が、証明書マネージャの使用可能な認証局証明書リストにある。2. 認証局証明書の発行が確認され、信頼できるものとして記されていること。

認証局証明書自体がない場合は、認証局証明書の証明書チェーンに、信頼できる高レベルの認証局証明書が含まれている必要があります。 また、証明書マネージャは、確認しようとしている証明書が証明書ストア内で「信頼されている」になっていることも確認します。これらのチェックに失敗すると、証明書マネージャは証明書を確認不可とし、その証明書が証明する識別情報を認識しません。

証明書はこれらのすべてのテストに合格しても、まだ認識されない場合があります。たとえば、権限のない人が証明書の秘密キーにアクセスした場合には証明書が破棄されることもあります。認識されない証明書では、権限のない人（または Web サイト）が証明書の所有者になりすますことができます。

証明書マネージャのこの問題を解決する 1 つの方法に、確認プロセスの一部として証明書破棄リスト（CRL）を確認する方法があります（以下の「CRL を管理する」を参照）。通常、CRL はリンクをクリックしてブラウザにダウンロードできます。 CRL がある場合、証明書マネージャは同じ認証局によって発行された証明書をこのリストと比較して確認します。

CRL の信頼性は、どのくらい頻繁にサーバが CRL を更新し、クライアントが確認するかによって変わってきます。 CRL の自動更新機能を設定すると、CRL は定期的にサーバ上の最新版に更新されます。

認識されない証明書に対処する別の方法として、OCSP（オンライン証明書ステータス プロトコル）をサポートする特別なサーバを使う方法があります。 このようなサーバは、個々の証明書に関するクライアントのクエリーに回答します（以下の「OCSP に対して証明書マネージャを設定する」を参照）。

OCSP レスポンダと呼ばれるサーバは、証明書を発行する認証局から、更新された CRL を定期的に受け取ります。OCSP レスポンダに証明書のステータス要求を送信するように証明書マネージャを設定すれば、OSCP レスポンダは証明書が有効かどうかを確認することができます。

 

CRL を管理する

証明書破棄リスト（CRL）は、破棄された証明書のリストです。認証局（CA）が証明書を無効にすることもあります。たとえば、証明書が認識されない場合です。これは、クレジット カード会社にクレジット カードの盗難届けを出すと、そのカードが無効になるのと同じような状況です。

このセクションでは、CRL をインポートおよび管理する方法について説明します。

詳しくは、「確認機能のしくみ」を参照してください。

管理可能な設定について詳しくは、「確認の設定」を参照してください。

「次の更新」の日付について

ブラウザは CRL を使用して、認証局が発行した証明書が有効かどうかを確認します。破棄された証明書としてリストされている場合、ブラウザはその証明書を身元証明として受け入れません。

通常、認証局は更新された CRL を定期的に公開します。各 CRL では、「次の更新」フィールドに日付が指定されています。これは、認証局がその CRL の次の更新を公開する日付です。 一般に、「次の更新」フィールドの日付が現在の日付より古い場合は、最新バージョンの CRL を取得する必要があります。 CRL に関する情報や CRL 自動更新の設定方法については、「CRL を表示、管理する」を参照してください。

認証局は次の更新日までに新しい CRL を作成する必要がありますが、最新版の CRL が存在しないために証明書が無効になることはありません。 このため、最新版の CRL がない場合には、最新版の CRL で期限切れと表示されている証明書をもっていても、その証明書は無効にはなりません。 CRL 自動更新機能を使用すると、このような状況を回避することができます。
